# Sync architecture (`src/hooks/useSync.ts`)

`useSync` is a single React hook mounted once at the app layout level. It owns all Supabase communication.

## Key module-level singletons

These live outside the hook and persist for the entire browser session:

- `lastPushedFsrs` — Map of `cardId → JSON` used for dirty tracking; prevents re-upserting unchanged FSRS rows on every debounce cycle. Seeded from server on each pull. Cleaned up when cards are deleted.
- `pushedLogIds` — Set of review log IDs already pushed. Cleared (then re-seeded) at the start of every full pull to prevent unbounded growth.
- `preExistingIds` / `preExistingSnapshotTaken` — Snapshot of localStorage IDs taken once per session before the first pull, used to distinguish "deleted on another device" from "created this session before push completed."

## `applyingRemoteRef` — the push-loop guard

**Rule: every `useLibraryStore.setState()` call that originates from a remote source (Supabase Realtime or BroadcastChannel) must be bracketed with `applyingRemoteRef.current = true/false`.**

The library store subscriber (`useLibraryStore.subscribe`) schedules a debounced push on any state change. Without the flag, every incoming realtime event echoes back as a phantom push of all folders/decks/cards — creating a self-sustaining loop with no user interaction. The flag is checked in the subscriber as an early return (`if (applyingRemoteRef.current) return`).

The exams and settings subscribers also check this flag. Notes have no Realtime subscription so they don't need it.

## Pull watermark

`sessionStorage` key `nemos-last-pull-at` stores the ISO timestamp of the last successful pull. Present → incremental pull (only rows changed since then). Absent → full pull (new tab or first load). `incremental` is the local variable that gates this throughout `runPull`.

## Cross-tab coordination

- **Web Lock** `'nemos-sync-push'` — serialises pushes across all open tabs. Auto-released when the lock callback resolves.
- **Web Lock** `'nemos-sync-leader'` — leader election. The tab that acquires it holds the lock (via a promise that never resolves) for its lifetime; `isLeaderRef` is `true` only in that tab. Auto-released on tab close/reload so another tab picks up leadership. Gates the periodic/visibility pull below so N open tabs don't all hit Supabase at once — non-leader tabs stay current via Realtime + BroadcastChannel instead.
- **BroadcastChannel** `'nemos-sync'` — tabs post messages after a push so peers can apply deletions and settings changes immediately rather than waiting for their next pull.

## Periodic / visibility pull

Besides the mount-time pull, the leader tab also calls `pullFromSupabase()` (incremental — it reuses the `nemos-last-pull-at` watermark, same as any other pull after the first) on a `visibilitychange` event (tab becomes visible again) and every 5 minutes via `setInterval` while active. This is what keeps a tab open all day from drifting out of sync with `fsrs_data` changes made on another device — the `review_logs` Realtime feed alone doesn't carry those scheduling updates.

# Deck Study popup (`src/components/library/StudyModePopup.tsx`)

Three modes: `deck-reviews`, `deck-new`, `deck-both`. All are deck-scoped and bypass the daily new-card limit (that limit is inbox-only).

## New-card count input

When the user selects **New Cards** or **Both** and the deck has at least one new card, the popup enters a two-step flow:
1. Mode button click → selects/highlights the mode and reveals a number input defaulting to `getDeckNewAll(deckId).length`.
2. **Start** button → navigates to `/study/session?deck=…&mode=…&newCount=N`.

Reviews mode still navigates immediately (no input). Both mode with zero new cards also navigates immediately.

The `?newCount=N` param is read in `session/page.tsx` as `deckNewCount`. In `buildQueue`:
- `deck-new`: slices `getDeckNewAll` to `deckNewCount`.
- `deck-both`: replicates the interleave logic inline (reviews unlimited, new cards capped at `deckNewCount`).
- `deck-reviews`: unaffected — no new-card count applies.

`deckNewCount` is in `buildQueue`'s `useCallback` dep array. Do **not** touch `newCardsPerDay`, `useSettingsStore`, or the daily new-card tracking — this feature is entirely separate from the automatic inbox limit.

### Fix: CAS-guard resurrection, stale reload watermark, dead realtime DELETEs, fsrs_data pull data-loss

Two independent data-integrity bugs in the sync layer, plus one cosmetic cleanup pass.

**Bug 1 — deletions could resurrect, in three compounding ways, all in `src/hooks/useSync.ts`:**

1. **CAS guard couldn't tell "new" from "deleted elsewhere" (`dropStaleOverwrites`, was line 549).** `if (!serverTs) return true` treated *any* row with no server copy as safe to push — including a row that used to exist server-side and was deleted on another device, which the guard then resurrected. Fix: `dropStaleOverwrites` now takes a `knownSyncedIds: Map<string, string>` parameter (the caller's `lastPushedFolders`/`lastPushedDecks`/`lastPushedCards`/`lastPushedNotes` dirty-tracking map) and checks it when `serverTs` is missing — present in the map (synced before, gone now) → drop the push; absent (never synced) → push normally. The dropped row's map entry is deliberately left alone (not deleted) so a later push cycle doesn't reclassify it as "new" and resurrect it after all.
2. **Cold-reload gap in that same check.** The `lastPushedX` maps are module-level and reset on every page load, so a plain reload's "not in map" no longer means "never synced" — it could just mean "synced last session, this session's map is empty." Traced the actual ordering: `handlePush` now hard-gates on `mountedRef.current` (new check added at the top of `handlePush`, mirroring the gate the store subscribers already had), and `mountedRef.current` is only set `true` after the mount-time pull's `.then()` fires — i.e. after a pull has already run and, per fix 2 below, that pull is *always* a full pull on a fresh load. A full pull's `runPull` (unchanged in this respect) seeds every `lastPushedX` map from every server row before `handlePush` can ever be called. So by the time any push runs, "not in the map" is trustworthy again — genuinely new, not stale.
3. **Stale `sessionStorage` watermark defeated the "full pull on reload" assumption fix 1.2 depends on.** `nemos-last-pull-at` survives a plain reload (sessionStorage only clears on tab/window close), so a reloaded tab was doing an *incremental* pull, which per `mergeIncremental`'s own doc comment can never represent a deletion. Fix: added a module-level `fullPullDoneThisLoad` boolean (line ~211) that starts `false` on every JS re-initialization (new tab **and** plain reload alike — unlike sessionStorage, a module-level var doesn't survive a reload). `pullFromSupabase` now does `const since = fullPullDoneThisLoad ? getLastPullAt() : null`, forcing the *first* pull of every page load to be full regardless of what the watermark says; `runPull` sets `fullPullDoneThisLoad = true` only after a successful **full** pull. Chose this over scoping the watermark to a random per-load instance id — same effect, one boolean instead of extra sessionStorage bookkeeping.
4. **Realtime DELETE events were never delivered at all — a rot deeper than the CAS guard.** Every `postgres_changes` subscription in the realtime effect (folders/decks/cards/review_logs, ~line 1384 in the pre-fix file) used a server-side `filter: 'user_id=eq.{userId}'`. Under default `REPLICA IDENTITY` (primary-key only, confirmed not overridden anywhere in `schema.sql`), a DELETE event's `old` payload contains **only the primary key** — no `user_id` — so a server-side filter on `user_id` can structurally never match a DELETE row, and Postgres/Realtime drops the event before it reaches the client. Separately (and worse), `schema.sql` never had `alter publication supabase_realtime add table ...` for *any* table — so realtime was fully inert for INSERT/UPDATE/DELETE alike, not just deletes. Fixed both: (a) removed the server-side `filter` from all four subscriptions and replaced it with client-side checks — INSERT/UPDATE payloads carry the full row, so an `isOwnRow(row)` helper compares `row.user_id` against the cached user id; DELETE payloads carry only the id, so ownership is inferred by checking whether that id exists in the local store (`useLibraryStore.getState().folders.some(f => f.id === id)`, etc.) — if it's not there, it's either not ours or already gone, safe no-op either way. This also fixed a related bug while in there: the cards-DELETE handler wasn't pruning the corresponding `fsrsData[cardId]` entry (the cross-tab BroadcastChannel delete handler already did this; the realtime one didn't — now it does). (b) Added the missing publication registration to `schema.sql` (new `do $$ ... alter publication supabase_realtime add table ...` block, idempotent via `pg_publication_tables` existence check) **and** a new standalone migration file for existing databases: **`src/lib/supabase/migration-add-realtime-publication.sql`** — must be run manually in the Supabase SQL Editor, same as `migration-drop-sm2.sql` before it.

**Bug 2 — `fsrs_data` pull merge was unconditional server-wins (`runPull`, was `{ ...current.fsrsData, ...fetchedFsrsData }`).** Any push-failure window (offline, tab closed mid-debounce, transient error) meant the *next* pull silently and permanently reverted local review progress to the stale server copy, and reseeded `lastPushedFsrs` from that stale copy so nothing even looked dirty for a retry — a real data-loss bug, not just a cosmetic one. Fix:
- `FSRSState` (`src/lib/srs.ts`) gained an optional `updatedAt?: string` field, stamped by `fsrsInitCard` at creation.
- `src/store/useLibraryStore.ts`'s `reviewCard` now stamps `updated.updatedAt = new Date().toISOString()` after `fsrsSchedule` runs (before: `fsrsSchedule` spreads the previous state forward, so without an explicit re-stamp a row pulled from the server would silently carry its old server `updatedAt` into the "new" review). `setFSRSData` (used by session-page undo to restore a pre-review snapshot) also stamps a fresh `updatedAt` on write, since restoring is itself a new local intent that must win the next merge and get re-pushed.
- `src/hooks/useSync.ts` gained `pickFresherFsrs(local, server)`: no local row → adopt server (pure new data, nothing to compare); local row with no `updatedAt` (legacy, pre-this-fix data) → server wins, same as the old behavior, closing itself out after the card's first review post-upgrade; both present → whichever `updatedAt` is later wins, ties go to server (post-push the server copy is content-identical and canonical, keeping `lastPushedFsrs` seeding consistent — a strictly-newer unpushed local review naturally stays dirty against the reseeded snapshot and retries on the next push cycle). Applied per-card-id in the `useLibraryStore.setState` merge (was previously a single object spread); whole-row comparison is correct here because `reviewCard` always writes a complete new `FSRSState` object per review, never partial fields.
- While tracing this path also fixed a related regression risk: the old code pre-filtered `fetchedFsrsData` to only cards present in *that pull's* `cardsRes.data` — which drops fsrs rows for cards whose `cards` row didn't change (the common case: a review on another device touches only `fsrs_data`, not `cards`). That pre-filter is now removed; orphan pruning (fsrs rows with no matching card) still happens correctly downstream against the full merged card set inside the `setState` updater, which is the right place for it since incremental pulls don't return the full card set.

- *New deck, never synced* — created this session, its id has never entered `lastPushedDecks` (nothing pushes without going through `dropStaleOverwrites` first, and that's the only place entries get added). On push, `serverUpdatedAt.get(r.id)` is `undefined` (no server row) and `knownSyncedIds.has(r.id)` is `false` → falls to `return true` → pushed normally. Correct.
- *Deck deleted elsewhere, this tab has a stale local copy* — on a prior successful push/pull this session, the id was written into `lastPushedDecks` (either by `runPull`'s post-fetch seeding loop or by the `lastPushedDecks.set(...)` after a successful upsert in `pushToSupabase`). Not resurrected. - *Cold-reload edge case* — reload happens; `lastPushedDecks` is now empty (module-level, doesn't survive reload); `fullPullDoneThisLoad` is also freshly `false`. `mountedRef.current` starts `false` and `handlePush` now returns immediately if so. So by the time a push is possible, the map already reflects "synced as of this full pull," and the deleted-elsewhere case above applies correctly even on a cold reload. No gap.
- *Realtime DELETE attribution without `user_id`* — a DELETE's `old` payload is PK-only (`{ id }`), confirmed by REPLICA IDENTITY DEFAULT semantics referenced in the schema. The new handlers check `useLibraryStore.getState().folders.some(f => f.id === id)` (etc., per table) instead of trusting a server-side filter that structurally can't apply to DELETE payloads. If the id isn't in the local store, either it was never ours (UUID collision across users is not a realistic concern) or it's already been removed — both cases make the early-return a safe no-op. Combined with the new publication registration (previously missing entirely, so no realtime events of any kind were flowing — confirmed by grepping `schema.sql` for `alter publication` before this fix and finding zero matches), this closes the dead-DELETE-delivery gap end to end: publication → event fires → client receives it → client-side id-membership check correctly attributes it without needing `user_id` in the payload at all.
- *fsrs_data merge, local newer* — device A reviews a card, push fails (offline). Device A's `fsrsData[id].updatedAt` is now `T2` (just stamped in `reviewCard`), server still has the pre-review row at `T1 < T2`. Device A's next pull runs `pickFresherFsrs(local={updatedAt:T2}, server={updatedAt:T1})` → `localTs(T2) > serverTs(T1)` → returns `local`. The review survives the pull. Because it was never removed from `current.fsrsData` (kept as-is by `pickFresherFsrs` returning the *local* object, not a merged/reset one) and `lastPushedFsrs` was never updated for this id (the failed push never reached the "record what we just pushed" step), the row is still dirty and the very next push cycle retries it. No data loss, no stuck state.
- *fsrs_data merge, server newer* — device A pushed a review successfully (`lastPushedFsrs` now has the matching snapshot, server `updated_at` trigger-stamped at push time). Device B (different device, unrelated local copy at an older `updatedAt`) pulls. `pickFresherFsrs(local=device B's stale row, server=device A's pushed row)` → server's `updatedAt` (set by the DB trigger at push time, necessarily ≥ device A's client stamp) is later → returns `server`. Device B correctly adopts the newer review. Standard last-write-wins, unchanged in spirit from every other table's pattern in this codebase.
- *fsrs_data merge, no local entry* — brand-new card pulled for the first time (e.g. created on another device). `pickFresherFsrs(undefined, server)` short-circuits on the first line (`if (!local) return server`) before touching either timestamp — no possibility of a spurious `NaN` comparison or accidental drop. Adopted unconditionally, as intended.

### Fix: fsrs_data integrity — 702 server-side orphans (cleanup migration) + cards restored without fsrs rows (backfill)

Two data-integrity bugs confirmed against a real export of `cards`/`fsrs_data` (702 of 1000 fsrs rows orphaned; 8 live cards with no fsrs row).

**Bug 1 — orphaned fsrs_data rows.** Every *current* delete path already cleans `fsrs_data` server-side — the sync pendingDeletes push (`useSync.ts`, fsrs delete per card chunk), the trash page's permanent-delete safety net (single-card and deck paths), and `deleteAllData` — but both earlier fixes were forward-only, and `runPull` prunes orphaned fsrs rows *locally* on every pull (filter against the merged card set) without ever queuing a server delete. So rows orphaned before those fixes (`due_date` cluster 2026-06-15→07-05, predating the earliest surviving card) were re-downloaded on every full pull, pruned locally, and never removed server-side. Client-side accumulation was ruled out: all four local delete actions prune `fsrsData`, the pull merge prunes to the merged card set in both branches, and no due-count/query logic iterates `fsrsData` keys — `getNewCards`/`getReviewsDue`/`getDueCards`/`getDeckNewAll`/`getDeckMastery` all iterate `cards` and look up `fsrsData`, so a client orphan can't inflate New/Due counts. **`src/lib/supabase/migration-cleanup-orphaned-fsrs.sql`** (new) is the retroactive half: a preview `select count(*)` plus `delete from fsrs_data where not exists (matching cards row, user_id-scoped)`. No FK was added to prevent recurrence: the push upserts `cards` and `fsrs_data` in a parallel `Promise.all`, so an FK would turn the fsrs-lands-first race into a push failure.

**Bug 2 — cards with no fsrs_data row.** `createCard`/`importCards` always call `fsrsInitCard`, but four *restore* paths re-added cards while silently skipping the FSRS entry whenever no snapshot existed: the trash page's card restore (`fsrsData: entry.cardFSRS ? … : s.fsrsData`), its deck restore (spread of `deckFSRS ?? {}`, and `deleteDeck`'s snapshot only includes cards that *had* entries, so gaps propagate through delete→restore), `deleteUndo.ts`'s `restoreCardsFromTrash` (`if (entry?.cardFSRS)`, plus the fallbackCards path which never carries FSRS), and `importFromJSON` (backups from before fsrs export carried no FSRS at all; legacy `srsData` conversion only covers cards that had an SM-2 entry). A card with a missing entry is treated as new everywhere (`getNewCards`: `!fs || fs.state === 'new'`; `getReviewsDue` excludes it; `reviewCard` heals it with `?? fsrsInitCard`, silently resetting history) — the "already-reviewed card shows up in New Cards" symptom. Fixes:

- **`src/lib/srs.ts`** — new `fsrsBackfillCard(cardId, userId)`: `fsrsInitCard` minus the `updatedAt` stamp. An unstamped row is deliberately the weakest claim in the sync layer, so a backfilled placeholder can never beat real scheduling data.
- **`src/app/(app)/trash/page.tsx`** / **`src/lib/deleteUndo.ts`** — all three trash/undo restore paths now guarantee an fsrs entry per restored card: snapshot ?? still-present store entry ?? `fsrsBackfillCard`.
- **`src/lib/import.ts`** — `importFromJSON` fills fsrs gaps for every card in the backup the same way (covers `restoreBackup`).
- **`src/hooks/useSync.ts`** — `runPull`'s merge now backfills an unstamped entry for every merged card missing one. This is what heals the 8 existing cards (and any device the gap propagated to): backfills are absent from `lastPushedFsrs`, so the next push sends them and the server row finally exists.
- **`src/hooks/useSync.ts`** — the fsrs push is partitioned: stamped rows take the normal upsert; unstamped rows (backfills + legacy pre-`updatedAt` rows) push with `ignoreDuplicates: true` (insert-only) so a placeholder can never overwrite a real server row — mirroring `pickFresherFsrs`, which already lets any server row beat an unstamped local one on pull. Without this, a device that pulled a card mid-push (before its fsrs row landed) could backfill and then clobber the real row.

**Manual migration required:** run `src/lib/supabase/migration-cleanup-orphaned-fsrs.sql` in the Supabase SQL Editor (preview select first, then the delete). The 8 missing rows need no SQL — the client backfill pushes them on the next sync.

Verified via `tsc --noEmit` and `next build` (both clean); `eslint` on every touched file shows only the two pre-documented pre-existing `useSync.ts` issues (mount-effect setState error + missing-deps warning).
